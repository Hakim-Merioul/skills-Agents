# GPT-M 2.0 Image Prompts

Used to generate PPT images for this skill in a Codex environment. Prompts set the tone only — do not write them as long explanations. First determine image placement and aspect ratio, then choose a type.

## General Rules

- First determine the current deck style: Style A = editorial magazine × e-ink; Style B = Swiss International Typographic Style
- Style A tone: editorial magazine × e-ink — restrained, authentic, generous whitespace, suited to horizontal web PPTs
- Style B tone: Swiss International Typographic Style — 12/16-column grid, Helvetica/Inter aesthetic, single high-saturation accent, square solid color, hairlines, extreme whitespace
- Text language in infographics, charts, and redesigned screenshots must follow the user's language: Chinese for Chinese decks, English for English decks
- No cartoons, 3D, neon tech, SaaS template feel, excessive decoration, or fake logos
- Images should leave space for titles or body text to overlay; do not fill the entire frame with fine detail
- All images in the same page or group must use the same aspect ratio, visual zoom, and margin density
- Images are embedded assets in a PPT, not standalone slides: do not generate headers, footers, page numbers, title bars, badges, attributions, decorative borders, or slide chrome
- After generating, save to `images/`, named `{slide-number}-{semantic}.{ext}`

## Aspect Ratio Selection

| Use | Recommended ratio | HTML placement |
|------|---------|-----------|
| Section cover / full-screen main visual | 16:9 | `.frame-img.r-16x9` or hero background reference |
| Swiss top banner / Image Hero | 16:9 or 21:9 | P22 top image cover / `.frame-img.r-21x9` |
| Left-text/right-image main image | 16:10 or 4:3 | `.frame-img.r-16x10` / `.frame-img.r-4x3` |
| Infographic / system diagram | 16:9 or 16:10 | Original screenshots use `.fit-contain`; regenerated to slot use `.frame-img.r-16x9` / `.frame-img.r-16x10` to fill |
| Redesigned screenshot / UI context image | 16:10 or 21:9 | Original screenshots use `.fit-contain`; regenerated for S15/S16 use `.frame-img.r-21x9` to fill |
| Small mixed image/text image | 3:2 or 3:4 | `.frame-img.r-3x2` / `.frame-img.r-3x4` |
| Image grid | Uniform horizontal | `.frame-img.h-22` / `.frame-img.h-26` |
| Small panel group | Uniform horizontal | `.frame-img.h-16` / `.frame-img.h-18` |

For infographics and redesigned screenshots from uncontrolled original assets, prefer `fit-contain` to avoid text being cropped. If images are regenerated by GPT-M 2.0 to the slot's ratio, generate to the exact slot ratio and fill the container — do not let a small image float in a white frame. Documentary photos prefer the default `cover` to maintain visual tension.

## Image Standardization Strategy

### A. Choose the target slot first

Do not generate an image first and then force it into a slide. Decide placement first:

1. Main visual: 16:9
2. Left-text/right-image: 16:10 or 4:3
3. Infographic/redesigned screenshot: 16:9 or 16:10, using `fit-contain`
4. Multi-image grid / panel group: uniform height class; mixing heights within a group is forbidden

### B. Handling user original images / screenshots

Original screenshot ratios are usually uncontrolled — do not treat them directly as the final visual standard. Process in this order:

1. If the original content needs fidelity, first read `screenshot-framing.md` and use the CleanShot-X-style programmatic approach: target-ratio canvas + styled background + proportionally scaled screenshot + semantic padding/alignment
2. If the original ratio is close to the target slot, place directly in a uniform `.frame-img` with `cover` or `fit-contain`
3. If a UI image stretches into a giant strip, split it into 2-3 same-size panels; each panel uses the same height class
4. If the original is too tall, too narrow, or too long and cannot be resolved through adaptation, regenerate with "screenshot redesign / UI context image" to the target ratio
5. If the original must be kept, use `fit-contain` inside a uniform frame; accept whitespace; do not crop critical text

### C. Generate prompt suffix

Append this constraint to every image prompt:

```text
Output must be a [16:9/16:10/4:3/3:2] horizontal composition, subject centered with margins retained, medium visual density, matching the same visual zoom and margin density as other images in the group. Keep only the core graphic/scene itself; do not generate headers, footers, headings, page numbers, badges, attributions, decorative borders, ultra-wide strips, portrait orientation, or irregular ratios.
```

When multiple images are needed on the same page, add:

```text
This is one image in a group; please maintain the same frame ratio, element size, margins, line thickness, and annotation density as the other images in the group.
```

## Style B: Swiss International Style Image Rules

When a deck uses `assets/template-swiss.html` / `layouts-swiss.md`, prioritize this set of prompts. They pair with GPT-M 2.0, targeting images that can go directly into the original registered layout slots — especially the S22 top banner and S15/S16 multi-image grids.

### Swiss Image Hard Rules

- Visual reference: International Typographic Style / Swiss modernism / Helvetica / Josef Müller-Brockmann / Massimo Vignelli
- Composition: strict 12/16-column grid, asymmetric whitespace, left-aligned, hairlines, square-cornered modules
- Color: black, white, grey only, plus **one** theme accent (IKB blue by default; substitute the corresponding accent if the user selects lemon yellow/green/safety orange)
- Forbidden: gradients, shadows, rounded corners, glassmorphism, neon, 3D, cartoons, SaaS template feel, fake logos, decorative borders
- Do not generate PPT shell inside the image: no headers, footers, page numbers, title bars, badges, attributions, outer frames
- UI / infographic text must be short; maintain consistent Chinese/English language; real photos should ideally be text-free
- Determine the layout slot before generating: single large image = `s22-hero-21x9`; multi-image grid = `s15-grid-21x9` or `s16-brief-21x9`
- 21:9 images must place the core subject within the central 70% safe zone, with whitespace on all sides; do not push faces, key nodes, or UI text to the edges
